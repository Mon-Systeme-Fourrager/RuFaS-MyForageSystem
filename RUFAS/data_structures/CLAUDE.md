# CLAUDE.md — RUFAS/data_structures/

Guidance for typed contracts between modules. See `RUFAS/CLAUDE.md` for module-level rules.

## Role

`data_structures/` defines the typed objects that cross module boundaries. They are the only legal way for `biophysical/` managers to exchange data with each other and with `routines/`.

## Naming Convention

Files and classes follow the `XToYConnection` pattern — named after the data flow direction:

| File | Flow |
|---|---|
| `feed_storage_to_animal_connection.py` | Feed data consumed by animals |
| `animal_to_manure_connection.py` | Manure generated by animals |
| `manure_to_crop_soil_connection.py` | Nutrients applied to fields |
| `crop_soil_to_feed_storage_connection.py` | Harvested crops entering storage |

For standalone domain types (not directional), use a plain descriptive name: `manure_nutrients.py`, `manure_types.py`.

## Which Type to Use

| Type | When to use |
|---|---|
| `@dataclass` | Structured data with multiple fields, needs `__post_init__` validation or methods |
| `@dataclass(frozen=True)` | Immutable snapshots (e.g., nutrient analysis results) |
| `Enum` | Categorical values used as dict keys or switch conditions (e.g., `StorageType`, `ManureType`) |
| `TypedDict` | Lightweight input shapes passed from config/JSON to constructors |

Prefer `@dataclass` over `TypedDict` when the object has any behavior (`__post_init__`, methods, properties).

## Rules

- No business logic in data structures. `__post_init__` may validate or derive fields from constructor inputs — nothing more.
- No imports from `biophysical/` or `routines/`. Data structures must be importable by both layers without circular dependency.
- Use `kw_only=True` on dataclasses with many fields to prevent positional argument mistakes.

## Adding a New Connection

1. Create `x_to_y_connection.py` in this directory.
2. Define the dataclass/TypedDict/Enum.
3. Import it in both the producing manager (`biophysical/x/`) and the consuming manager (`biophysical/y/`).
4. Add a test in `tests/` validating the structure (instantiation, validation, edge cases).
